Modelling Process vs Feature Engineering

FEAT ไม่ใช่ขั้นตอนที่แยกจาก ML modelling process แต่เป็นส่วนหนึ่งที่สำคัญของกระบวนการ

กระบวนการย่อยของ FEAT เกี่ยวข้อง/เจาะลึกในกระบวนการที่เกี่ยวข้องกับการจัดการข้อมูลโดยเฉพาะ อาจจำแนกเป็น 4 ขั้นตอน ได้แก่

  • Feature Understanding – วิเคราะห์ทำความเข้าใจคุณลักษณะของ feature แต่ละตัว เช่น การระบุสถานะของข้อมูล ความหมายเบืิ้องต้น การแจกแจงเบื้องต้น (ถ้าสามารถทำได้) และความสัมพันธ์กับตัวแปรเป้าหมาย

  • Feature Structuring – แปลงข้อมูลที่ไม่มีโครงสร้าง เช่น ข้อความ รูปภาพ เสียง และอื่น ๆ ให้อยู่ในรูปแบบที่มีโครงสร้าง เพื่อให้ ML model สามารถนำไปใช้ได้

  • Feature Optimization – ใช้เทคนิคต่าง ๆ ปรับแต่ง/ปรับปรุง feature ที่มีอยู่ หรือสร้าง feature ใหม่จากข้อมูลเดิม เพื่อให้ข้อมูลมีสัญญาณหลักที่ชัดเจนขึ้น มีความหมายที่ตีความได้ง่าย สมเหตุสมผล และเพิ่มประสิทธิภาพการทำนายของโมเดล

  • Feature Evaluation – ประเมินคุณภาพของ feature ที่สร้างขึ้นหรือปรับแต่งแล้ว โดยใช้ metric ต่าง ๆ เช่น feature importance, correlation analysis, mutual information, predictive power analysis เป็นต้น

ประเภทของ Feature Engineering

  • Feature Improvement – การปรับปรุงคุณภาพของ feature ที่มีอยู่ เช่น การจัดการกับค่าที่ขาดหาย การลด noise การปรับสเกล และการแปลงข้อมูล

  • Feature Construction – การสร้าง feature ใหม่จากข้อมูลเดิม เช่น การสร้างตัวแปรเชิงปริมาณจากตัวแปรเชิงหมวดหมู่ การสร้างตัวแปร interaction และการสร้างดัชนีต่าง ๆ

  • Feature Extraction – อัลกอริทึมสำหรับดึงสาระสำคัญ/ข้อมูลที่ต้องการ จากข้อมูลดิบออกมาเป็น feature เช่น PCA, autoencoders หรือ text embeddings

  • Feature Learning – การใช้โมเดลการเรียนรู้ของเครื่อง เช่น neural networks เพื่อเรียนรู้และสร้าง feature ใหม่จากข้อมูลดิบ

Feature Engineering

ประสิทธิภาพโมเดลมักได้มาจาก คุณภาพฟีเจอร์ มากกว่าความซับซ้อนของอัลกอริทึม

  • Feature Engineering คือกระบวนการแปลงหรือปรับแต่งข้อมูลดิบ (raw data) ให้อยู่ในรูปแบบที่เหมาะสมสำหรับการสร้างโมเดลการเรียนรู้ของเครื่อง

    • เพิ่มประสิทธิภาพการเรียนรู้ของโมเดล –> ประสิทธิภาพในการทำนาย

    • แก้ไขปัญหาลดข้อจำกัดที่เกิดขึ้นในข้อมูล

    • ช่วยให้การตีความหมายผลลัพธ์ของโมเดลทำได้ง่ายขึ้น

  • Feature Engineering = ศาสตร์/ศิลป์ แปลงข้อมูลดิบให้กลายเป็น features ที่เครื่องสามารถเรียนรู้ได้ดี

  • แยกเป็น 2 มิติ

    • Data Preprocessing (ให้โมเดลทำงานได้)

    • Feature Creation (ทำให้โมเดลทำงานได้ดี)

ควรทำ feature engineering ตอนไหน?

  1. ก่อนที่จะ train model

  2. ระหว่างที่ train model

  3. หลังจากที่ train model แล้ว

แนวคิดหลัก

ประเภท เป้าหมายหลัก ตัวอย่าง step_* (คละ basic/advance) เมื่อควรใช้ / ข้อควรระวัง
Data Preprocessing
(ทำให้โมเดล “ทำงานได้”)		 แก้ NA, แปลงชนิดข้อมูล, ปรับสเกล/รูปแจกแจง, ตัด noise/redundancy step_impute_knn, step_impute_bag, step_unknown, step_novel • step_other, step_dummy • step_log, step_YeoJohnson, step_BoxCox • step_normalize, step_spatialsign • step_zv, step_nzv, step_corr, step_lincomb • step_date, step_holiday • themis::step_smote¹, themis::step_downsample, themis::step_upsample - ใช้ prep() กับ train เท่านั้น เพื่อกัน leakage; sampling steps (เช่น step_smote) ใส่ skip=TRUE เพื่อ ไม่ ใช้กับข้อมูลใหม่.
- ใส่ลำดับให้ถูก: encode → scale → filter; ตัวอย่าง interaction ควรเกิด หลัง dummy
Feature Creation
(ทำให้โมเดล “ทำงานได้ดีขึ้น”)		 สร้างสัญญาณใหม่ จับ nonlinearity/interaction โครงสร้างเวลา/ข้อความ step_interact • step_poly, step_ns • step_discretize • step_mutate (row-wise ratios/logs) • step_pca, step_kpca, step_ica • embed::step_umap² • embed::step_lencode_glm, embed::step_lencode_mixed² (impact/target-based encoding ภายใต้การควบคุม) • textrecipess::step_tokenize, textrecipess::step_stopwords, textrecipess::step_ngram, textrecipess::step_tfidf³ - เลี่ยง leakage เมื่อใช้ encoding อิงเป้าหมาย: ทำภายใน resampling ที่ถูกต้องเท่านั้น (ผ่าน fit_resamples).
- step_mutate ให้คงเป็น row-wise (เช่น x/y, log(x)); อย่าคำนวณสถิติทั้งคอลัมน์ใน bake()
Boundary / Hybrid ลดยุ่งเหยิง/มิติ พร้อมคงสัญญาณ step_other (ก่อน dummy), step_corr (ลด redundancy) • step_pca (หลัง normalize) • embed::step_umap² - ชั่งน้ำหนักระหว่าง ตีความได้ (filter selection) กับ พลังทำนาย (feature extraction).
- ตรวจสอบผลด้วย metric ทั้งเชิงพยากรณ์และความอธิบายได้.

เชิงอ้างอิง/บันทึก
1) themis เหมาะกับ class imbalance (step_smote, step_downsample, step_upsample); ใส่ไว้ ก่อน โมเดลและใช้เฉพาะ train fold (skip=TRUE).
2) embed ให้เครื่องมือ encoding/embedding ขั้นสูง (step_lencode_*, step_umap).
3) textrecipess ใช้กับข้อมูลข้อความทั้งสาย (tokenize → clean → n-gram → tf-idf ฯลฯ).

Data Leakage และการป้องกันด้วย prep()

Caution

เมื่อเราใช้ prep() กับ recipe สุดท้าย
ระบบจะ fit recipe เฉพาะกับ training set (ตามที่ระบุใน recipe())
ซึ่งช่วยป้องกันปัญหา data leakage ได้!

Data Leakage คืออะไร?

  • ข้อมูลจาก นอก training set (เช่น test set) รั่วเข้ามาในขั้นตอน train model
  • ส่งผลให้โมเดลดูเหมือนทำงานได้ดีเกินจริง เพราะมันได้เห็น test set บางส่วนมาแล้ว
  • มักเกิดขึ้นเมื่อมีการคำนวณค่าทางสถิติจาก ทั้ง dataset แทนที่จะใช้เฉพาะ training set

ผลกระทบจาก Data Leakage

  • โมเดลมี performance สูงเกินจริงบน test set
  • แต่เมื่อเจอ ข้อมูลใหม่จริง ๆ ประสิทธิภาพจะตกลง
  • สร้างความเข้าใจผิดในการประเมินโมเดล

Note: Imperatives vs Declarative Programming

Imperative Programming
- เขียนคำสั่ง → รันทันที
- ได้ผลลัพธ์ออกมาในตอนนั้นเลย
- สิ่งที่ผู้ใช้ R ส่วนใหญ่คุ้นเคย เช่น

x <- scale(mydata$x)

Declarative Programming
- เขียนคำสั่งแบบ “ประกาศ” ว่า จะทำอะไร พร้อมข้อกำหนด (constraints)
- การรันจริงจะเกิดขึ้น “ทีหลัง” ตามที่ผู้ใช้หรือโปรแกรมกำหนด
- เช่น การสร้าง recipe ใน tidymodels

rec <- recipe(y ~ ., data = train) |>
    step_normalize(all_numeric_predictors())

1.1 Missing Value

ปัญหา missing value เป็นปัญหาที่พบบ่อยและอาจสร้างผลกระทบอย่างมากต่อโมเดลการเรียนรู้ของเครื่อง การจัดการ missing value มีหลายวิธีการ แต่ละวิธีการจะมีข้อดี ข้อจำกัดและเหมาะกับบริบทการใช้งานที่แตกต่างกัน เราอาจจำแนกบริบทการใช้งานได้เป็น 2 ลักษณะ

  • การแก้ปัญหา missing ใน inferential statistics

    • เน้นอธิบาย เน้นอนุมานไปยังประชากร/กลุ่มเป้าหมาย

    • พยายามรักษาความน่าเชื่อถือ/ความถูกต้องของการอนุมาน

    • เน้นวิธีการ impute missing ที่สามารถ capture ความไม่แน่นอนของการทดแทนค่าสูญหายให้ได้ เช่น MI

    • ปัจจัยสำคัญที่ใช้เลือกวิธีการ impute คือ กลไกการสูญหายของข้อมูล

  • การแก้ปัญหา missing ใน predictive modeling

    • เน้นสร้างโมเดลที่มีความแม่นยำ

    • การจัดการ missing มีความสำคัญเพียงในขั้นตอน preprocessing เพื่อสร้างโมเดลทำนายที่ดีที่สุด

    • เนื่องจากเน้นผลการทำนายที่ดี วิธีการที่ใช้จึงเน้นการทดแทนค่าสูญหายที่แม่นยำและทำให้โมเดลมีค่าทำนายที่แม่นยำที่สุด

    • การอนุมานหรืออธิบายความสัมพันธ์ในประชากรไม่ใช่วัตถุประสงค์หลัก

1.2 Scaling/Normalization

  • Normalization เป็นวิธีการปรับตัวแปรโดยนำค่าของมันมาลบด้วยค่าเฉลี่ย (mean) แล้วหารด้วยส่วนเบี่ยงเบนมาตรฐาน (standard deviation)

  • ผลลัพธ์ที่ได้จะเป็นสเกลใหม่ที่มีค่าเฉลี่ยเป็น 0 และส่วนเบี่ยงเบนมาตรฐานเป็น 1

  • อย่าลืมว่าการแปลงนี้จะคำนวณค่าเฉลี่ยและส่วนเบี่ยงเบนมาตรฐานจาก training data เท่านั้น แล้วจึงนำค่าที่ได้ไปใช้แปลงทั้ง training และ test data เพื่อหลีกเลี่ยงปัญหา data leakage

  • นอกจากนี้ยังมี misconcept ที่ว่าการแปลงแบบนี้ช่วยให้ข้อมูลมีการแจกแจงแบบปกติ

\[ X_{scaled} = \frac{X - mean(X)}{SD(X)} \] ## Normalization {.smaller}

recipe(academic_risk ~ ., data = train_data) %>% 
  update_role(student_id, new_role = "ID") %>%
  step_rm(gpa) %>% 
  step_normalize(distance_from_home)

1.2 Scaling/Normalization : Range Scaling

อีกชื่อหนึ่งคือ Min-Max Scaling เป็นวิธีการที่ช่วยแปลงตัวแปรเชิงปริมาณให้มีสเกลอยู่ในช่วง [0,1]

\[ X_{scaled} = \frac{X - min(X)}{max(X) - min(X)} \]

recipe(academic_risk ~ ., data = train_data) %>% 
  update_role(student_id, new_role = "ID") %>%
  step_rm(gpa) %>% 
  step_range(distance_from_home,
             min = 0,
             max = 0.5)

1.2 Scaling/Normalization: Robust Scaling

  • เป็นวิธีการแปลงที่คล้ายการทำ normalization แต่ใช้ค่ามัธยฐาน (median) และช่วง interquartile range (IQR) แทนค่าเฉลี่ยและส่วนเบี่ยงเบนมาตรฐาน

  • อย่างไรก็ตามวิธีตัวหาร IQR(X) อาจมีค่าเท่ากับ 0 ในบางชุดข้อมูล

\[ X_{scaled} = \frac{X - median(X)}{IQR(X)} \]

  • มีฟังก์ชันสำเร็จรูปใน library-extrasteps

  • โดยปกติการแจกแจงที่ไม่เบ้มากเกินไปของตัวแปรทำนาย เพียงพอที่จะทำให้การเรียนรู้ของเครื่องทำงานได้อย่างมีประสิทธิภาพ (เพราะหลายเทคนิคไม่ได้ require normality ในตัวแปรทำนาย) แต่บางกรณีก็มีเงื่อนไข normality ในตัวแปรทำนาย เช่น linear discriminant analysis (LDA)

  • Box-Cox transformation เป็นเทคนิคที่ใช้แปลงตัวแปรเชิงปริมาณให้มีการแจกแจงใกล้เคียง normal distribution มากขึ้น

  • หลักการทำงานคือ ใช้ maximum likelihood estimation (MLE) เพื่อประมาณค่าพารามิเตอร์การแปลง () ในสมการที่กำหนด เพื่อหาการแปลงที่ทำให้ข้อมูลใกล้เคียงการแจกแจงแบบปกติมากที่สุด เราต้องประมาณค่าพารามิเตอร์ จาก training data set ก่อน แล้วจึงนำค่าที่ได้ไปใช้แปลงทั้ง training และ test data set เพื่อหลีกเลี่ยงปัญหา data leakage

  • เงื่อนไขที่สำคัญของ Box-Cox transformation คือ ตัวแปรต้องมีค่าบวกเท่านั้น (positive values)

              limits = c(-5,5), # A length 2 numeric vector defining the range to compute the transformation parameter lambda.
              lambdas = NULL) # A numeric vector of transformation values. This is NULL until computed by prep().

Percential Scaling

  • บางครั้งเรียกว่า Rank Scaling หรือ Quantile Scaling

  • เป็นวิธีการแปลงข้อมูลแบบไม่เป็นเชิงเส้น (non-linear transformation) โดยที่ค่าของแต่ละจุดข้อมูลจะถูกแทนที่ด้วย เปอร์เซ็นไทล์ (percentile) เมื่อเทียบกับข้อมูลในชุด training data

Note

คำว่า percentile และ quantile ใช้แทนสิ่งเดียวกัน เพียงแต่ต่างกันที่วิธีการแสดงผล percentile แสดงเป็นเปอร์เซ็นต์ เช่น 10%, 20%, 30% ส่วน quantile แสดงเป็นทศนิยม เช่น 0.1, 0.2, 0.3

  • ผลลัพธ์ที่ได้จากการแปลงจะได้สเกลใหม่ที่มีค่าอยู่ในช่วง [0,1]

  • ระมัดระวังดี ๆ ว่าข้อมูลชุดอื่นที่จะนำมาใช้งานเช่น test data หรือ new data ควรต้องมีการแจกแจงในลักษณะใกล้เคียงกับชุดข้อมูล traning data

1.4 Categorical Encoding

  • ข้อมูลอีกประเภทที่พบบ่อยคือ ตัวแปรจัดประเภท (categorical variables) ซึ่งเป็นตัวแปรที่ไม่สามารถวัดค่าได้เป็นตัวเลข แต่มีลักษณะเป็นกลุ่มหรือประเภท

  • โมเดลหลายประเภท เช่น linear regression, logistic regression, SVM, k-NN, neural networks ต้องการให้ข้อมูลเป็นตัวเลขทั้งหมด จึงจำเป็นต้องแปลงตัวแปรจัดประเภทให้เป็นตัวแปรเชิงปริมาณก่อนนำไปใช้สร้างโมเดล

  • โมเดลบางประเภท โดยเฉพาะกลุ่ม โมเดลเชิงต้นไม้ (tree-based models) เช่น decision trees, random forests, gradient boosting machines สามารถจัดการกับตัวแปรจัดประเภทได้โดยตรง โดยไม่ต้องแปลงเป็นตัวเลขก่อน

1.4 Categorical Encoding

มีประเด็นที่ควรพิจารณา ได้แก่

  • ค่าที่เป็นไปได้ของตัวแปรจัดประเภท – ข้อมูลของตัวแปรจัดประเภทสามารถมีจำนวนค่าที่เป็นไปได้ทั้งแบบจำกัดค่าแน่นอน เช่น เพศ วันใน 1 สัปดาห์ หรือ วิธีการสอนที่ครูใช้ หรือ อาจมีค่าที่เป็นไปได้แบบไม่จำกัด/ไม่แน่นอน เช่น ชื่อโรงเรียน ชื่อผลิตภัณฑ์ …

  • ลำดับของข้อมูล (ordering) – ตัวแปรจัดประเภทบางตัวมีลำดับขั้นของข้อมูลในธรรมชาติ เช่น ขนาดโรงเรียน ระดับชั้นของนักเรียน

  • การเข้ารหัส (encoding) – วิธีการเข้ารหัสที่แตกต่างกัน อาจมีผลต่อประสิทธิภาพของโมเดล

1.4 Categorical Encoding

  • Dummy-based

    • Dummy encoding – step_dummy() แปลง level ของ factor ให้เป็นตัวแปร 0, 1

    • Label encoding – step_integer() แปลง level ของ factor ให้เป็นตัวเลขลำดับ

  • Target-based

    • step_lencode_glm() เป็นฟังก์ชันในกลุ่ม Supervised Factor Conversion มีหน้าที่แปลง Factor ให้เป็นตัวเลขโดยใช้โมเดล Generalized Linear Model (GLM) เราอาจอธิบายหลักการทำงานคร่าว ๆ ได้ดังนี้ (1) สร้าง glm ที่ทำนาย outcome ของโมเดลกับ predictor ที่เป็น factor (2) ใช้ค่าสัมประสิทธิ์ของโมเดลที่ได้จาก glm เป็นคะแนนของระดับปัจจัยแต่ละระดับ

    • step_lencode_bayes() หลักการเดียวกับ glm แต่เปลี่ยนการประมาณค่าพารามิเตอร์ในโมเดลเป็น bayesian glm ผลลัพธ์ที่ได้จะมีความแกร่งว่าการใช้ glm แบบปกติโดยเฉพาะกรณีขนาดตัวอย่างเล็ก ความแตกต่างอีกส่วนนึงที่น่าสนใจคือ step_lencode_bayes() จะให้ solution ที่แตกต่างกันไปในแต่ละครั้ง เนื่องจากการประมาณค่าพารามิเตอร์ใช้ Markov Chain Monte Carlo (MCMC) ที่อิงกับตัวเลขสุ่ม

    • step_lencode_mixed() หลักการเดียวกับ glm แต่ใช้ mixed-effects model แทน เหมาะกับข้อมูลที่มีโครงสร้างแบบ hierarchical หรือ panel data

  • Other methods

    • step_num2factor() แปลงตัวแปรเชิงปริมาณเป็นตัวแปรจัดประเภท

    • step_unorder() แปลงตัวแปรจัดประเภทที่มีลำดับขั้นเป็นตัวแปรจัดประเภทที่ไม่มีลำดับขั้น

    • step_other() รวม level ที่มีความถี่ต่ำเข้าด้วยกันเป็นกลุ่มใหม่เรียกว่า “other”

    • step_unknown() แทนที่ค่าที่ไม่รู้จัก (unknown) หรือ ค่าที่หายไป (missing) ด้วย level ใหม่ที่ชื่อว่า “unknown”

    • step_novel() แทนที่ค่าที่ไม่เคยพบใน training data ด้วย level ใหม่ที่ชื่อว่า “novel”

2. Feature Construction

Feature Construction คือการสร้างฟีเจอร์ใหม่ขึ้นมาโดยเจตนา จากฟีเจอร์เดิม เพื่อ ใส่ความรู้เชิงแนวคิด (conceptual / domain knowledge) เข้าไปในข้อมูล ให้โมเดลเห็นโครงสร้างของปัญหาได้ชัดกว่าที่ข้อมูลดิบเล่าเองได้

  • สร้างตัวแปรใหม่โดยใช้ทฤษฎี ตรรกะ หรือความรู้ที่เกี่ยวข้องกับบริบทการทำงาน

  • จัดกลุ่มข้อมูล (binning/discretization) เป็นการแปลงข้อมูลเชิงปริมาณให้เป็นข้อมูลเชิงหมวดหมู่ เหมาะกับกรณีที่ความสัมพันธ์ของข้อมูลไม่ได้เป็นเส้นตรง หรือกรณีที่มีค่าผิดปกติในตัวแปร

step_discretize() ## แบ่งช่วงตาม quantile พยายามให้แต่ละ bin มีจำนวนข้อมูลเท่ากัน
step_cut() ## แบ่งช่วงตาม cut points ที่กำหนดเอง

2.2 Binning

  • เป็นเทคนิคการแปลงข้อมูลเชิงปริมาณให้เป็นข้อมูลเชิงหมวดหมู่ (categorical) โดยการแบ่งช่วงค่าของตัวแปรเชิงปริมาณออกเป็นกลุ่มย่อย (bins) ตามเกณฑ์ที่กำหนด จากนั้นแทนค่าแต่ละช่วงด้วยตัวแปรใหม่

  • วิธีการนี้ช่วยให้สามารถจัดการความสัมพันธ์ระหว่างตัวแปรตามกับตัวแปรทำนายในลักษณะที่ไม่เป็นเชิงเส้นตรงได้

  • ลดความซ้ำซ้อน
